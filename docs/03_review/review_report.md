# ARY GRS 001 一致性审查报告

## 1. 审查结论

当前文档整体通过 P7 第一部分一致性审查。

方案主线不是普通 PRD，而是围绕“Organizer 数据主权 + ARY 不持久化完整 Race 数据 + Public Projection 展示”的最小可证明设计展开。PRD、PoC、架构、数据流和 Demo 在核心口径上保持一致。

## 2. 主要发现

### R-001 DCR 边界已收敛，但仍需在代码 PoC 阶段验证

- **严重度**：中。
- **位置**：`architecture.md`、`data_flow.md`、`decision_log.md`。
- **发现**：架构已明确 DCR 接触完整 Race 数据时必须位于 Organizer 授权域内，ARY 侧只处理公开数据。该边界合理，但仍是待验证设计，真实实现中 DCR 的部署位置、运行身份和输出内容需要被检查。
- **风险**：如果代码 PoC 中 DCR 输出中间结果到 ARY，可能形成隐性完整数据复制。
- **建议**：代码 PoC 阶段必须加入 DCR 输出清单检查。

### R-002 Public Projection 字段过宽仍是主要待验证风险

- **严重度**：中。
- **位置**：`poc_validation_matrix.md`、`architecture.md`、`data_flow.md`、`demo_plan.md`。
- **发现**：文档已多次标记 Public Projection 不是完整数据源，并要求字段清单可审计。但完整 Race 数据分类尚未最终确定，投影字段过宽仍可能削弱“去中心化数据主权”的证明力度。
- **风险**：投影若接近完整数据，ARY 虽未直接存完整数据，但展示链路仍可能泄露过多内容。
- **建议**：代码 PoC 前先冻结最小投影字段集合。

### R-003 个别文档状态存在轻微滞后表述

- **严重度**：低。
- **位置**：`prd.md`、`poc_validation_matrix.md`。
- **发现**：`prd.md` 自检中仍写 DCR 边界“留待 P4”，而 P4 已形成架构工作假设；`poc_validation_matrix.md` 中仍写“P4 前形成最小字段分类”，但 P4 已完成且该项仍未最终冻结。
- **风险**：不影响核心方案，但提交前可能让评审误以为阶段状态未同步。
- **建议**：P7 提交整理阶段统一修正文档状态措辞。

## 4. 是否写成普通 PRD

结论：否。

依据：

- PRD 没有扩展为普通赛事管理系统。
- Must 功能围绕创建、披露、组织、展示和状态处理。
- PoC、架构、Demo 均围绕四项核心证明对象。
- 非目标明确排除指导骑行、成果评审、骑行复盘、生产级全栈能力。

## 5. 数据安全与去中心化证明检查

| 证明对象 | 结论 | 证据 |
| --- | --- | --- |
| 数据留存 | 通过 | PoC 使用 Organizer 完整数据、ARY 登记清单和展示字段清单做对比 |
| 去中心化 | 基本通过 | ARY Registry 限定最小登记信息；代码 PoC 需验证字段未膨胀 |
| 功能闭环 | 通过 | PRD、PoC、Demo 均覆盖创建、披露、组织、展示 |
| 投影机制 | 基本通过 | 展示字段可溯源到 Public Projection；投影字段范围仍需冻结 |

## 6. 隐性持久化完整 Race 数据检查

当前未发现必须把完整 Race 数据复制到 ARY 的设计链路。

仍需重点防守的路径：

- ARY Registry 字段膨胀。
- Public Projection 字段过宽。
- DCR 输出中间结果到 ARY。
- ARY Display 或缓存长期保存投影正文。
- 验证证据保存私有内容正文。

这些风险已在 `architecture.md`、`data_flow.md`、`decision_log.md` 中标记，未发现未标记的重大隐性复制路径。

## 7. 角色职责冲突检查

| 角色 | 审查结论 |
| --- | --- |
| Organizer | 职责一致：拥有完整数据，决定披露、更新、撤回 |
| ARY | 职责一致：登记公开入口、组织公开 Race、按需展示投影 |
| DCR | 基本一致：作为执行能力，不是数据所有者；接触完整数据时需在 Organizer 授权域 |
| Public Projection | 职责一致：公开有限视图，不是完整数据源 |

未发现职责直接冲突。
