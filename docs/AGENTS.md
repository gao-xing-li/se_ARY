# ARY 项目工作约束

## 1. 项目目标

- 围绕 ARY GRS 001，定义“数据主权归 Organizer、ARY 不持久化完整 Race 数据”条件下的赛事产品方案。
- 交付重点包括：PRD、PoC 理论与验证方案、系统架构与数据流、Demo 展示方案、Riding Record。
- 以问题定义、证明力度、架构边界和协作过程可复盘性作为质量核心。

## 2. 事实来源

- 首要依据：`docs/00_sources/source_ARY_GRS_full_text.md`。
- PDF 仅用于核对首要依据中的缺失、歧义或排版信息。
- 文档必须区分：来源事实、设计决策、待验证假设、待确认问题。
- 不得将未确认推测写成既定需求。

## 3. 不可突破的设计边界

- Race 完整数据仅留存在 Organizer 侧。
- ARY 不得中心化持久化保存完整 Race 数据。
- ARY 展示内容必须来自 Organizer 主动披露的公开元数据或公开投影。
- 方案必须说明赛事创建、披露、组织与展示如何形成闭环。
- Organizer、ARY、DCR、Public Projection 的职责、数据权限和持久化边界必须明确且无越权。

## 4. 工作原则

- 先明确问题、边界和验收标准，再开展产品与技术设计。
- 采用最小可证明方案，不扩展到赛事指导、成果评审、骑行复盘等非核心子系统。
- 每项关键选择记录到 `docs/02_design/decision_log.md`，包含依据、影响和状态。
- 发现歧义时先记录待确认问题，不静默补全。
- 阶段切换以 `PLAN.md` 为准；未明确进入实现阶段前，不编写实际 PoC 代码。

## 5. Agent 执行规范

- 动手前显式说明假设、歧义和关键权衡；无法安全判断时先询问。
- 优先采用解决问题的最简单方案，不增加未要求的功能、抽象或配置。
- 只修改任务直接需要的内容，不重构或清理无关内容。
- 将任务拆成可验证目标，完成后按验收标准检查并修正。

## 6. 文档规范

- 所有项目文档使用中文，内容关键、精炼。
- 图示必须标明组件边界、数据方向、数据所有者和持久化位置。
- 方案必须能够追溯到四项 PoC 核心证明：数据留存、去中心化、功能闭环、投影机制。
- Riding Record 持续记录目标、Agent 指令与输出、人类判断、修改原因和验收结果。
- 只修改当前任务直接需要的文件，不进行无关重构或扩展。

## 7. 完成标准

- 产品逻辑能够解释数据不离开 Organizer 时的完整赛事闭环。
- 理论 PoC 为每项核心假设提供可执行的验证步骤与通过标准。
- 架构与数据流不存在隐含的完整数据复制或中心化持久化。
- Demo 方案能让评审直观看懂数据主权、公开投影和功能闭环。
- 设计过程可追踪、可解释、可复盘。
