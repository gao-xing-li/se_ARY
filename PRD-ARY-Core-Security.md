# ARY 产品需求文档 (PRD) - 核心安全与组织者隐私保护

**版本:** 1.0
**核心主题:** 保障 Organizer (组织者) 数据的绝对安全边界与加密确权

---

## 1. 产品愿景与定位 (Vision & Positioning)

ARY 是一款专为“组织者（Organizer）”设计的极高安全性数据管理与协作平台。

在传统的 SaaS 工具（如 CRM、活动管理软件、表格协作工具）中，服务商拥有最高权限，可以轻易通过数据库或后端服务访问用户的明文业务数据。ARY 采用**“零知识 (Zero-Knowledge)”与“端到端加密 (E2EE)”架构**，彻底颠覆这一信任模型。

**核心定位：** 保障 Organizer 产生和管理的任何敏感数据（如高净值人员名单、财务流转、核心行程安排、私密通讯）对平台侧（包括开发人员、DBA、甚至被攻陷的服务器）做到物理隔离级别的绝对不可见。平台只提供数据的加密存储、流转和加密检索计算服务，**不“拥有”数据**。

## 2. 核心用户画像 (Target Personas)

*   **高净值/敏感活动组织者 (High-Profile Event Organizers)：** 组织高端峰会、私人聚会、政府或企业高级别会议，需要绝对保密参会者的真实身份和敏感行程安排。
*   **DAO / 去中心化社区核心贡献者 (Community Leaders)：** 管理社区金库或核心成员的 KYC 数据，不能容忍单点数据库泄露带来的灾难性信任危机。
*   **私密企业/研发团队管理者：** 需要在团队核心成员内部进行高度机密的任务分配、专利探讨和资源流转，防范内部威胁和商业间谍。

## 3. 产品边界 (Product Boundaries)

明确产品边界是维持极高安全基准的前提。任何可能引入侧信道攻击（Side-Channel Attack）或削弱端到端加密体系的“便利性功能”，都必须坚决排除在外。

### 3.1 明确在边界内 (Strictly In-Scope)
1.  **零信任身份验证 (Zero-Trust IAM)：** 强依赖设备级认证（如 Passkey/WebAuthn 生物识别），消除传统弱密码被撞库或钓鱼的风险。
2.  **全量客户端加密 (Client-Side Encryption, CSE)：** 核心业务数据在离开 Organizer 设备前，必须在浏览器/客户端内存中完成高强度加密。
3.  **加密状态下的数据检索 (Searchable Encryption)：** 解决“数据全加密后无法搜索”的难题，允许 Organizer 在不向服务器暴露查询明文的情况下检索数据（如采用盲索引 Blind Index 技术）。
4.  **基于密码学的访问控制 (Cryptographic Access Control)：** Organizer 间的数据分享必须通过非对称加密的密钥交换完成，而绝不能仅仅通过修改服务器数据库的 `permission_role` 字段来实现。
5.  **不可篡改的操作审计日志 (WORM Audit Trail)：** 所有针对极敏感数据的访问、解密、授权变更需记录，并确保证据级防篡改（可引入类似 Merkle Tree 或轻量区块链固化）。

### 3.2 坚决在边界外 (Strictly Out-of-Scope)
1.  **公域分发与 SEO 优化公开页：** 绝不提供免密、无需身份验证的公开访问链接（如传统的“活动报名 H5”或“公开可被搜索引擎抓取的日程表”）。ARY 是私密保险箱，不是扩音喇叭。
2.  **基于服务器的明文数据分析图表：** 不提供任何依赖服务器计算的业务数据统计图表（例如：由服务器聚合出的“按地区分布的参与者统计”）。因为服务器无法解密数据，所有图表聚合必须在客户端获取密文、解密后在本地内存中渲染。
3.  **弱验证的第三方集成 (Standard Webhook/OpenAPI)：** 轻易开放 API 会成为数据明文泄露的巨大后门。初期坚决不提供，或仅提供经过端到端加密验证的有限 API 通道。
4.  **传统的邮件/短信营销广播 (EDM/SMS Blast)：** 发送明文短信/邮件意味着必须将明文明文交由第三方通道处理，这严重违背了 E2EE 原则。平台默认仅提供端内的强加密通知。

## 4. 核心功能模块划分 (Core Modules)

### 4.1 终端设备与密钥金库 (Secure Device & Keystore)
*   **设备绑定与隔离：** 每次登录严格绑定当前硬件设备，主私钥由系统安全模块（如 Secure Enclave / TPM）或强密码保护的本地 IndexedDB 存储，**私钥明文永不上云**。
*   **恢复短语 (Recovery Phrase)：** 类似数字加密货币钱包，提供 12/24 词的助记词，用于设备丢失时的账号和数据恢复，确保没有“找回密码”的系统级后门。

### 4.2 隐秘数据对象引擎 (The Vault Engine)
*   **结构化密文存储：** 支持 Organizer 创建自定义的数据表（如“VVIP 参与者名单”），表格的 Schema（表头结构）和 Data（行数据）对服务器均是加密的。
*   **字段级细粒度安全策略：** 允许 Organizer 指定哪些是“普通字段”（加密但可被平台弱索引，如“注册时间”），哪些是“极度敏感字段”（强加密，仅支持精准查询，如“真实姓名”、“护照号”、“资金流水”）。

### 4.3 密码学授权与协作协作 (Cryptographic Collaboration)
*   **零信任数据分享：** Organizer A 将数据分享给 B 时，A 的客户端会自动获取 B 之前公开的公钥，将数据的对称加密密钥（DEK）用 B 的公钥加密后，存回服务器。B 登录后用自己的私钥解密，进而获取该条数据。服务器无法在此过程中窃取授权。
*   **防泄漏视图：** 在前端实现渲染层的控制，包括限制选中复制、添加针对当前操作人身份的隐形动态水印、设置“阅后即焚”时间锁。

## 5. 关键技术 PoC (概念验证) 方案设计

**PoC 核心目标：** 
构建一个最小化原型，验证在“假设后端服务器完全不可信或被黑客攻破”的前提下，平台仍能顺畅实现数据的**加密存储**、**精准共享**，以及最关键的技术难点——**对密文数据的高效搜索**。

### 5.1 痛点与深度技术选型
传统的 E2EE 方案会导致**丧失服务器端的数据检索能力**。如果 Organizer 存了 10,000 条参与者名单，想要搜索 "Alice"，难道要把 10,000 条密文全部拉取到客户端解密后再搜索？这将导致无法接受的网络开销和性能灾难。
**PoC 验证核心：引入“盲索引 (Blind Index)”结合信封加密 (Envelope Encryption) 技术。**

### 5.2 PoC 架构与关键链路设计

**角色定义:**
*   **Alice (Organizer)：** 数据拥有者。
*   **Bob (受邀协作者)：** 被 Alice 信任并授权的人。
*   **Evil Server (平台后端/DBA)：** 负责存储与路由，但不能知道任何业务明文。

#### 阶段 1：密钥生成与初始化 (Key Generation)
1.  Alice 在浏览器端注册。前端 `Web Crypto API` 生成：
    *   **非对称密钥对 (Asymmetric Keypair):** `Alice_Pub` (公钥，上传服务器公开), `Alice_Priv` (私钥，加密后本地 IndexedDB 存储)。
    *   **盲索引主密钥 (Index Key, IK):** 对称密钥，用于生成搜索特征码，随私钥一起本地安全保存。

#### 阶段 2：数据的加密与存储 (Encryption & Storage)
1.  Alice 要添加一条高敏记录：`{"name": "John Doe", "phone": "13800000000"}`。
2.  前端随机生成一个专门针对该记录的**数据加密密钥 (DEK)**，使用 `AES-256-GCM` 算法加密这段 JSON，得到密文 `Ciphertext`。
3.  前端使用 `Alice_Pub` 加密这个 DEK，得到 `Encrypted_DEK_Alice`。
4.  **盲索引生成 (关键步骤)：** 为了让服务器能基于名字搜索 "John Doe"，前端使用 IK (Index Key) 计算 "John Doe" 的 HMAC：
    *   `Blind_Index_Name = HMAC-SHA256(IK, "johndoe") = "a8f5c9..."`
5.  前端将最终的 Payload 发送给服务器：
    `{ "ciphertext": "...", "encrypted_DEK": {"alice_uid": "Encrypted_DEK_Alice"}, "blind_indices": {"name": "a8f5c9..."} }`
6.  **验证点：** 此时登录服务器数据库（模拟黑客视角），看到的只有 AES 乱码和无意义的 HMAC Hash，完全无法推断出这行数据代表 "John Doe"。

#### 阶段 3：零知识密文检索 (Zero-Knowledge Search)
1.  Alice 在前端搜索框输入 "John Doe"。
2.  前端使用本地提取的 `IK`，对输入内容计算同样的 HMAC：`Search_Token = HMAC-SHA256(IK, "johndoe")`。
3.  前端向服务器发起请求：`GET /api/records?blind_index_name=Search_Token`。
4.  服务器执行简单的数据库 `WHERE blind_index_name = ?` 精确匹配（或类似 Redis 的哈希查找），将命中的 `Ciphertext` 和 `Encrypted_DEK_Alice` 返回给客户端。
5.  客户端使用 `Alice_Priv` 解密出 DEK，再用 DEK 解密出真实的 `{"name": "John Doe"...}` 数据并在 UI 渲染。
6.  **验证点：** 服务器完美完成了检索服务，且数据库检索效率极高（走普通数据库索引），但服务器依然处于“零知识”状态，不知道用户搜索的明文是什么。

#### 阶段 4：数据安全共享 (Secure Sharing to Bob)
1.  Alice 决定将此特定数据共享给 Bob。
2.  Alice 客户端从服务器获取 Bob 的公钥 `Bob_Pub`。
3.  Alice 客户端利用 `Alice_Priv` 还原出该数据的 DEK。
4.  Alice 客户端使用 `Bob_Pub` 重新加密此 DEK，并将新的授权包裹追加存入服务器：
    `{ ..., "encrypted_DEK": {"alice_uid": "...", "bob_uid": "Encrypted_DEK_Bob"} }`
5.  *(进阶思考)* 为了让 Bob 也能具备搜索能力，Alice 必须通过安全通道（如用 Bob 的公钥加密）将盲索引主密钥 `IK` 共享给 Bob，让 Bob 客户端也能计算出相同的 HMAC。
6.  **验证点：** Bob 登录后成功检索并解密数据。在整个授权转移和使用过程中，服务器仅仅充当了“加密信封搬运工”的角色。

### 5.3 PoC 交付物与验证标准
为了验证上述技术可行性并展示给利益相关者，PoC 将交付以下三部分：
1.  **DB Admin View (黑暗视角验证)：** 一个直接连库的管理后台界面，直观展示数据库中的真实落盘数据，证明所有核心数据字段和索引均是一堆乱码，不具备可读性。
2.  **Network Inspector (网络抓包验证)：** 在前端展示实时的网络请求日志，证明发送给后端的 payload 和后端返回的 payload，绝对不包含未加密的敏感字符串。
3.  **Organizer & Collaborator UI (业务流转验证)：** 提供一个极简的 Web 应用界面，演示 Alice 添加高密数据 -> 搜索加密数据 -> 一键通过公钥体系安全授权给 Bob -> Bob 成功读取的完整闭环流程体验，验证其与传统 SaaS 的交互延迟在可接受范围内。
