# ARY GRS 001 创世骑行记录 (Riding Record)

**核心理念：** 人不是把活完全丢给 Agent，而是在骑行、检查、纠偏。在 Agent 时代，工程师更像是一名“骑手”，驾驭着强大的 AI 引擎，在探索中不断修正航向。

## 骑行轨迹 (The Ride)

本次 ARY GRS 001 PoC 的开发过程，是一次典型的人机协同探索：

1. **确立边界 (奠基阶段)：** 最初的指令并非直接写代码，而是要求 Agent 扮演产品经理与架构师，确立“去中心化数据主权”的铁律。通过定义严格的 JSON Schema 边界，为后续开发定下基调。
2. **构建引擎 (实现阶段)：** 引导 Agent 使用 FastAPI 快速将架构转化为单文件 PoC。在设计报名数据流时，通过引入 `Stateless Proxy` 模式，确保 ARY 平台只透传、不存盘。
3. **视觉与旅程 (包装阶段)：** 赋予 Agent “UI 设计师”角色，要求其输出具备“赛博朋克+玻璃拟态”的高级感页面；同时要求补齐完整的 API 旅程，让 PoC 从静态走向动态。
4. **关键纠偏 (重新对齐)：** 在开发深入时发现，早期将“报名信息”视为最高机密导致了功能受限。作为骑手，我果断介入并下达“重要纠偏”指令：明确真正的核心机密是“提交的代码、执行日志、复盘材料等事实库”，而公开的报名计数是可以披露的。这一指令重塑了系统对核心隐私的定义。

---

## 核心复盘闭环

**1. 认知从模糊到清晰的迭代**
一开始的 PRD 对数据隔离的定义过于一刀切，试图把所有的东西都藏在 Organizer 侧。但在让 Agent 编写实际的 Proxy 和渲染页面的过程中，我们发现如果连基本的报名数量都不可见，平台的“公开展示”就失去了意义。正是这种“边做边看”的骑行模式，逼迫我们重新思考到底什么是真正的“核心隐私资产”（执行代码与执行日志），从而完成了从概念到可落地架构的认知升级。

**2. 骑手的“方向盘”作用**
Agent 拥有极其强大的代码生成和重构能力，但它缺乏对业务本质的动态判断力。在 Prompt 8 到 Prompt 9 的转折中，如果没有人类敏锐地察觉到“隐私对象定义跑偏”并强制纠偏，Agent 会继续沿着错误的假设构建出不符合实际应用场景的系统。骑手的核心价值在于随时感知偏离，并通过精准的 Prompt 踩刹车、打方向盘。

**3. 人机协同的化学反应**
人类提供边界感、产品直觉和关键纠偏；Agent 提供不知疲倦的结构化输出、视觉实现和代码框架。这种组合让原本需要数天才能跑通的概念验证（从 PRD 到架构图再到全栈 PoC），在几轮高质量的对话闭环中迅速成型。这不是单纯的外包，而是高频互动的共创。

---

## 反思

在智能体时代进行软件工程，不再是“我把需求写好，你帮我写代码”的单向外包模式，而是**“我提出假设，你快速具象化，我们共同看着具象化的结果来修正假设”**的双向互动。代码生成只是副产品，真正的产出是在高频纠偏中逐渐清晰的产品边界。不要奢求“一发入魂”的完美 Prompt，**要学会享受紧握缰绳、在高速狂飙中不断微调方向的“骑行感”**。
## 我在 Agent 输出后完成的纠偏

在 ARY GRS 001 的设计骑行过程中，Agent 早期倾向于把报名信息、RiderID 和报名计数都当成核心隐私资产。作为人类 Rider，我判断这不符合 ARY GRS 001 的真实产品重点。

我重新定义了核心保护对象：参赛者代码、完整骑行记录、执行日志、DCR 判断链、评审证据、复盘材料和私有规则细节。这些才是必须留在 Organizer / DCR 控制域内的 Race Source Facts。报名计数、公开 RiderID / nickname、公开参与摘要，只要由 Organizer 主动披露，就可以作为 Public Projection 展示。

随后我要求 Agent 调整 PoC 的边界：更新 `FORBIDDEN_ARY_KEYS`，允许 Public Projection 包含公开报名摘要，修正 `debug/privacy-check`，并更新公开页面中的说明文案。这体现了 Agent Riding 的闭环：观察输出、发现偏差、进行人工产品判断、重新提示、验证修正结果。

## Final Verification Loop

最后一轮工作不是继续堆叠新功能，而是把 PoC 拉回验收闭环。我要求检查端口是否一致，确认创建公开对象与披露公开投影的旅程可以跑通，并验证 ARY 的泄露检查能够同时拒绝 key-level 和 value-level 的私有源事实。

我还要求验证 projection version 与 projection hash 的规则：旧版本必须被拒绝，同版本同 hash 可以幂等提交，同版本不同 hash 必须返回冲突，防止公开投影内容漂移。报名表单可以经过 ARY 代理，但长期展示的公开报名摘要仍必须来自 Organizer 主动披露的 Public Projection。

这一轮体现的是验收闭环：不是只看页面是否漂亮，也不是只看接口是否能返回，而是确认产品边界、数据主权、负向拒绝、公开披露和用户交互是否共同成立。

最后一轮不是继续增加功能，而是清理会影响现场演示和满分观感的风险。我检查并修正了端口一致性、表单依赖风险、泄露检查、Projection 版本/hash 规则，以及公开页面和调试接口的验收路径。

这说明本次 Agent Riding 没有停在“生成了一个看起来可用的产物”，而是继续进入验收阶段：发现演示风险、收紧说明、补充负向验证、确认边界表达，最终形成从生成到验收的闭环。
